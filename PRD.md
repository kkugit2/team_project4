# 채용 정보 플랫폼 PRD (Product Requirements Document)

## 0. 프로젝트 개요

**채용 정보 플랫폼**은 구직자와 기업 인사팀을 연결하는 양방향 채용 생태계 플랫폼입니다. 구직자는 자신의 경력 정보와 원티드 오픈 API의 실제 채용공고 데이터를 기반으로, "기업이 원하는 조건을 얼마나 충족하는지 진단 + 자기계발 방향 제시", 그리고 AI 기반 자소서 분석을 통해 지원 전략을 수립할 수 있습니다. 기업 인사팀은 원하는 인재상을 사전 등록하고, 구직자들의 자소서 정보를 대시보드에서 일치도 순으로 조회하여 **"인재상에 부합하는 지원자 발굴 + 스카웃 채널"**을 할 수 있습니다. 이를 통해 구직자와 기업 모두에게 최적화된 채용 매칭 경험을 제공합니다.

---

## 1. 왜 만드는가 (문제 정의)
- 구직자는 자신의 이력이 시장에서 어느 정도 위치인지, 무엇을 보완해야 하는지 객관적으로 알기 어렵다.
- 기업은 방대한 지원자 중 자사 인재상에 맞는 사람을 찾는 데 시간이 많이 든다.
- 원티드 API는 실제 채용공고/기업 데이터를 제공하지만, "분석·추천·매칭 근거"까지는 제공하지 않는다. → 이 서비스가 그 간극을 채운다.

## 2. 타깃 사용자 분석

### 페르소나 1: 구직자 (주요 사용층)
- **특징**: 대학교 졸업자, 신입 또는 경력직 구직자 (22~40세)
- **주요 니즈**:
  - 자신의 경력으로 합격할 수 있는 기업들을 빠르게 찾고 싶음
  - 특정 기업에 지원하기 전에 합격 확률을 알고 싶음
  - 자신의 자소서가 얼마나 경쟁력 있는지 평가받고 싶음
  - 같은 직무의 다른 구직자와 비교하여 부족한 부분을 파악하고 싶음
  - 지원 현황과 찜한 공고를 한곳에서 관리하고 싶음

### 페르소나 2: 기업 인사팀 (채용 담당자)
- **특징**: 기업의 인사/채용 담당자, 영입 담당자 (30~50세)
- **주요 니즈**:
  - 자신의 기업에 필요한 인재상을 사전에 등록하고 싶음
  - 우리 회사에 맞는 인재를 효율적으로 발굴하고 싶음
  - 구직자의 자소서와 경력을 빠르게 검토하고 싶음
  - 우리 기업에 지원한 사람이 누구인지 파악하고 싶음
  - 자격 요건과의 일치도를 정량적으로 비교하고 싶음

## 3. 핵심 기능 목록

### 3.1 구직자 기능

| 기능 | 설명 |
|------|------|
| **로그인/회원가입** | Supabase 인증 활용. 회원가입 시 아이디, 비밀번호, 학적, 학점, 자격증, 경력 정보 입력 (모든 항목 필수이지만 '해당 사항 없음' 선택 가능) |
| **랜딩페이지/채용공고 대시보드** | 여러 기업의 채용공고를 카드 형식으로 표시. 각 카드에는 기업명, 직급, 요구사항 충족률, 찜 상태, 지원 현황 표시 |
   ### 주요 구성 요소
   | 요소 | 설명 | 데이터 출처 |
   |---|---|---|
   | 공고 카드 리스트 | 회사 로고, 회사명, 직군, 마감일 | 원티드 포지션(Job) API |
   | **요구사항 충족률 배지** | "요구사항 4개 중 3개 충족 (75%)" 형태로 표시 | 자체 로직 (섹션 1-1) |
   | 지원현황 태그 | "지원 / 미지원" 표시 | Supabase `applications` |
   | 찜 아이콘 | 하트 토글 | Supabase `bookmarks` |
   | 필터/정렬 | 직군, 마감임박순, 충족률순 | 클라이언트 사이드 필터링 |
| **합격 확률 예측** | 우리 자체 데이터(기업 회원가입 시 입력한 인재상)와 구직자 정보를 규칙 기반으로 비교한 "요구사항 충족률(%)"을 "합격 확률(%)"로 제시 (MVP 데모에서는 구직자·기업 데이터 모두 AI로 생성한 더미 데이터를 활용) |
| **기업 상세 정보 조회** | 대시보드의 카드 클릭 시 기업 소개, 채용 포지션, 기업 정보 조회 (원티드 API 활용) |
| **지원하기 연결** | 기업 상세 페이지의 '지원하기' 버튼 클릭 시 해당 기업의 실제 지원서 작성 사이트로 외부 링크 이동. '지원하기' 버튼 옆에 '지원 완료' 버튼을 별도로 배치하여, 실제 지원을 마친 구직자가 직접 클릭해 지원 상태를 변경 |
| **채용공고 찜하기** | 관심 있는 채용공고를 찜하고, 찜한 목록 조회 및 관리 (DB에 저장) |
| **지원현황 추적** | 지원한 기업 및 공고의 현황을 표시 ('지원'/'미지원' 중 하나의 상태로 표시됨. '지원 완료' 버튼 클릭 시 수동으로 상태가 '미지원'에서 '지원'으로 변경됨) |
| **마이페이지** | 학적, 학점, 자격증, 경력 정보 조회/수정/삭제. 프로필 정보 변경 시 합격 확률 재산정 |
| **자소서 입력 및 AI 분석** | 자소서 텍스트 입력 후 외부 LLM API 기반 자동 분석. 자체 데이터 기반 피드백(강점, 약점, 개선 방안), 경쟁자 비교분석 제공 |

**예외 케이스**
- 비로그인 접근 → 합격확률/지원현황/찜 없이 공고 리스트만, 로그인 유도 배너
- 경력 정보 미입력 회원 → 배지 대신 "마이페이지에서 경력을 입력하면 합격확률을 볼 수 있어요" 안내

### 3-1-1. 요구사항 충족률 산출 방식

기업이 회원가입 시 입력한 인재상(`company_profile`)과 구직자의 `seeker_profile`을 1:1로 비교하는 **완전 규칙 기반 계산**이다. 외부 API나 AI 예측 없이 즉시 구현 가능.

```
충족 항목 수 = 0
전체 항목 수 = 기업이 실제로 설정한 조건 개수 (최대 4개: 학점, 기술스택, 유사경험, 인턴십 여부)
※ 기업이 온보딩에서 '건너뛰기'를 선택해 특정 조건을 설정하지 않은 경우, 해당 조건은 전체 항목 수에서 제외된다
   (즉, 지원자 조회 시 그 항목은 매칭 점수에 영향을 주지 않는다)

- 학점: company.preferred_gpa_min이 설정되어 있고 seeker.gpa >= company.preferred_gpa_min → 충족
  (company.preferred_gpa_min 미설정 시 이 항목은 계산에서 제외)
- 기술스택: company.preferred_skills가 1개 이상 설정되어 있고 seeker.skills 와의 교집합이 1개 이상 → 충족
  (company.preferred_skills 미설정 시 이 항목은 계산에서 제외)
  (※ 이때 스킬명 표기가 서로 달라 매칭 실패하는 걸 막기 위해,
     원티드 태그(Tag) API의 스킬 마스터 리스트를 두 폼 모두에서
     동일한 드롭다운/자동완성으로 쓰도록 강제한다)
- 유사경험: company.preferred_experience_type이 설정되어 있고 seeker.career_history 중 하나라도 일치 → 충족
  (company.preferred_experience_type 미설정 시 이 항목은 계산에서 제외)
- 인턴십 경험: company.internship_required가 true인데 seeker 경력에 인턴 항목이 있으면 충족
  (기업이 인턴십을 요구하지 않으면 이 항목은 계산에서 제외)

충족률(%) = (충족 항목 수 / 전체 항목 수) * 100
```

### 3.2 기업 인사팀 기능

| 기능 | 설명 |
|------|------|
| **로그인/회원가입** | Supabase 인증. 회원가입 시 기업/구직자 선택. 기업은 원하는 인재상 저장 (학점, 기술스택, 유사 경험, 인턴십 경험 등) |
| **인재상 설정 및 관리** | 회원가입 또는 마이페이지에서 원하는 인재상 조건 입력/수정 (학점 범위, 필수 기술스택, 선호 경력 등) |
| **구직자 자소서 대시보드** | 구직자들의 자소서와 경력 정보를 카드 형식으로 조회. 기업의 인재상과의 일치도 점수 표시 |
   **목적**: 기업 담당자가 정보 제공에 동의한 구직자를 인재상 부합도 순으로 확인

   | 요소 | 설명 | 데이터 출처 |
   |---|---|---|
   | 지원자 카드 리스트 | 익명화된 프로필 요약 (정보 제공 동의자만) | Supabase `self_intros` (`shared_with_company=true`) |
   | 인재상 부합도 배지 | 기업이 입력한 인재상과 비교한 점수 | 자체 매칭 로직 |
   | 지원 여부 표시 | 해당 구직자의 자사 지원 여부 | Supabase `applications` (job_id 필터) |
   | 스카웃 버튼 | 관심 인재에게 제안 발송 | Supabase `scouts` (3-2-1 참고) |
   | **인재 일치도 분석** | 기업이 등록한 인재상과 구직자의 경력/자격을 비교 분석하여 일치도 점수 계산 및 표시 (0~100%) |
   | **지원 여부 확인** | 각 구직자가 자신의 기업에 지원했는지 여부 표시 |
   | **상세 구직자 정보 조회** | 구직자 카드 클릭 시 해당 구직자의 상세 경력, 자격증, 자소서 정보 조회 |

**3-2-1. 스카웃 장치 (기업 실익 구조 — 확정)**
> 이전 논의에서 "기업이 확인만 가능하고 실질적 컨택 수단이 없다"는 문제가 제기되었고, 스카웃 기능을 추가하는 것으로 결정됨.
- 스카웃 제안 시 **기업명·포지션 공개**, 연봉/조건은 선택 공개
- 구직자 반응: **수락 / 거절 / 무응답**(7일 후 자동 만료)
- 기업당 **월 발송 건수 제한**(기본값 10건, 서버단 강제) — 스팸 방지
- 스카웃 = "관심 표시"이며 프리패스 아님. 수락 시 정식 지원 페이지로 연결, 기업 쪽엔 "스카웃 수락자"로 별도 표시되어 우선 검토 유도

## 4. 사용자 플로우

### 4.1 구직자 플로우

#### 플로우 1: 초기 사용자 (회원가입 및 첫 채용공고 탐색)

1. **회원가입**
   - 랜딩페이지에서 '회원가입' 선택
   - 사용자 유형 선택 화면에서 '구직자' 선택
   - 회원가입 폼에서 필수 정보 입력 (아이디, 비밀번호, 학적, 학점, 자격증, 경력)
   - 계정 생성 완료

2. **로그인**
   - 아이디와 비밀번호로 로그인
   - 로그인 성공 시 랜딩페이지로 이동

3. **채용공고 대시보드 확인**
   - 여러 기업의 채용공고가 카드 형식으로 표시
   - 각 카드에서 확인 가능한 정보: 기업명, 직급, 합격 확률(%), 찜 아이콘, 지원 현황
   - 카드를 스크롤하거나 필터로 조회

4. **기업 상세 정보 조회**
   - 공고 카드를 클릭하면 기업 상세 페이지로 이동
   - 기업 소개, 근무지, 복리후생, 채용 포지션 정보 확인
   - '지원하기' 버튼 클릭 시 해당 기업 지원 사이트로 새 창 열기
   - 실제 지원을 마쳤다면 '지원 완료' 버튼을 클릭해 지원 상태를 '지원'으로 변경

5. **찜하기**
   - 공고 카드 또는 상세 페이지의 하트 아이콘 클릭
   - 찜 추가 또는 제거
   - 찜한 공고 목록에서 별도로 조회 가능

#### 플로우 2: 마이페이지 - 프로필 관리

1. **마이페이지 진입**
   - 헤더의 '마이페이지' 링크 클릭
   - 현재 프로필 정보 조회 (학적, 학점, 자격증, 경력)

2. **정보 수정**
   - 각 항목의 '수정' 버튼 클릭
   - 수정 폼 표시
   - 변경 사항 저장 → DB 업데이트
   - 프로필 정보 변경 시 대시보드의 합격 확률 자동 재산정

3. 지원서 작성 후 피드백 페이지
**입력**: 자기소개서 텍스트, 목표 공고 선택

| 항목 | 산출 방식 |
|---|---|
| 피드백 | 자소서 + 공고(`intro`,`main_tasks`,`requirements`)를 LLM에 함께 입력해 "부합하는 점/보완할 점" 생성 |
| 나의 장단점 | 이력서(경력·자격증) + 자소서를 LLM으로 구조화 분석 |
| 경쟁자 비교분석 | 같은 공고에 제출한 다른 구직자(익명화 요약 통계)와 비교. **표본 5명 미만이면 "비교 데이터 부족" 안내로 대체** (무의미한 비교·개인식별 위험 방지) |

### 4.2 기업 인사팀 플로우

#### 플로우 1: 회원가입 및 인재상 설정

1. **회원가입**
   - 랜딩페이지에서 '회원가입' 선택
   - 사용자 유형 선택에서 '기업' 선택
   - 회원가입 폼: 회사명, 회사 이메일, 비밀번호 입력
   - 원하는 인재상 기본 조건 입력 (학점 범위, 필수 기술스택, 선호 경력, 인턴십 경험 등)
   - 이메일/비밀번호로 Supabase Auth 가입
   - 온보딩 폼에서 위 인재상 세부 조건(선호 학점 범위, 필수 기술스택, 선호 경력, 인턴십 경험 요구 여부)을 다시 확인·보완 입력 ("건너뛰기" 허용. 특정 조건을 입력하지 않으면 해당 조건은 충족률 계산에서 제외되며, 지원자 조회 시 그 항목의 영향은 사라짐 — 예: 선호 학점 범위를 입력하지 않으면 학점은 매칭 기준에 반영되지 않음)

2. **로그인**
   - 회사 이메일과 비밀번호로 로그인
   - 대시보드로 이동

#### 플로우 2: 구직자 인재 발굴

1. **구직자 대시보드**
   - 구직자들의 자소서와 경력 정보가 카드 형식으로 표시
   - 각 카드에서 확인 가능한 정보:
     - 구직자 이름/학력
     - 경력 요약
     - 인재상 일치도 점수 (0~100%, 색상 코딩)
     - 자신의 기업에 지원 여부
   - 일치도 높은 순으로 정렬 (기본값) 또는 필터링 가능

2. **상세 구직자 정보 조회**
   - 카드를 클릭하면 구직자 상세 페이지로 이동
   - 상세 정보: 학적, 학점, 자격증, 경력 이력, 자소서
   - 인재상과의 일치도 상세 분석 표시

3. **마이페이지 (구직자용)**
| 섹션 | 내용 | 데이터 출처 |
|---|---|---|
| 프로필 요약 | 학적, 학점, 자격증, 경력 | Supabase (회원가입 값) |
| 정보 수정 | 회원가입 폼 필드 재사용 | Supabase update |
| 나의 지원현황 | 지원한 공고 리스트 (자진신고) | `applications` + 원티드 `/jobs/{id}` 병합 |
| 찜 목록 | 찜한 공고 리스트 | `bookmarks` + `/jobs/{id}` 병합 |
| 스카웃함 | 받은 스카웃 제안 목록, 수락/거절 처리 | `scouts` |

### 5.2 테이블 상세 정의

#### 1. **User** (사용자 - 구직자 또는 기업)
| 컬럼명 | 타입 | 설명 | 제약 |

| created_at | TIMESTAMP | 계정 생성 시간 | DEFAULT CURRENT_TIMESTAMP |
| updated_at | TIMESTAMP | 마지막 수정 시간 | DEFAULT CURRENT_TIMESTAMP |

| id | UUID | 프로필 ID | PK |

### 5.3 관계 설명
- **1:1 관계**:
  - User ↔ JobSeekerProfile (구직자 1명 = 프로필 1개)
  - User ↔ RecruiterProfile (기업 1개 = 프로필 1개)
  
- **1:N 관계**:
  - JobSeekerProfile ↔ Certification (구직자 1명 = 자격증 여러 개)
  - JobSeekerProfile ↔ Experience (구직자 1명 = 경력 여러 개)
  - User ↔ bookmarks (구직자 1명 = 찜한 공고 여러 개)
  - User ↔ self_intros (구직자 1명 = 자소서 여러 개)
  - User ↔ applications (구직자 1명 = 지원 현황 여러 개)

## 7. 성공 기준 (데모 기준)

> 본 PRD에서는 기능별 P0/P1/P2 표기를 별도로 두지 않으며, 아래 성공 기준이 MVP 범위와 우선순위를 판단하는 실질적 기준이다.

- 구직자 플로우(회원가입 → 자소서 분석 → 추천 확인 → 지원 기록) end-to-end 동작
- 기업 플로우(회원가입 → 인재상 등록 → 랭킹 확인 → 스카웃 발송 → 수락/거절 반영) end-to-end 동작
- 실제 원티드 API 데이터가 최소 1개 화면 이상에 반영

## 8. 유의사항 및 제약조건

### API 사용 유의사항
- **원티드 OpenAPI 인증**: API 키 관리 및 Rate Limiting 고려
- **데이터 동기화**: 원티드 API의 채용공고 데이터 주기적 동기화 필요

### 보안 고려사항
- **비밀번호 관리**: Supabase의 내장 보안 기능 활용
- **HTTPS**: 모든 통신 암호화
- **개인정보보호**: GDPR, CCPA 등 규정 준수
- **AI 데이터**: 자소서 데이터 안전성 및 개인정보 보호

### 성능 고려사항
- **캐싱**: 원티드 API 응답 캐싱
- **일치도 계산**: 대량 구직자 데이터에 대한 일치도 계산 최적화
- **데이터베이스 인덱싱**: 자주 조회하는 컬럼에 인덱스 추가
