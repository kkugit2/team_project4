## 0. 서비스 한 줄 요약

원티드 오픈 API의 실제 채용공고 데이터를 기반으로, 구직자에게는 "합격 가능성 진단 + 자기계발 방향 제시"를, 기업에게는 "인재상에 부합하는 지원자 발굴"을 제공하는 양면 플랫폼.

**핵심 제약 (모든 화면 설계의 전제)**
- 원티드 API는 **조회 전용**이다. 실제 지원서 제출 API가 없으므로, "지원하기"는 항상 외부 사이트로의 이동이다.
- 연봉·경쟁률·지원자 수는 공개 API에 없다. 있는 것은 회사 단위 평균연봉(`/insight/company`, 사업자번호 필요)뿐이다.
- 합격확률은 원티드의 유료 AI 예측 API(`/ai/pass/text-prediction`)를 쓰거나, 자체 매칭 로직(스킬/자격증/경력 비교)으로 대체한다. 이 문서에서는 두 가지 경로를 모두 열어두고 설계한다.
- 회원 데이터(경력, 자격증, 지원현황, 찜 목록, 자소서)는 전부 Supabase에 우리가 직접 저장·관리한다.

## 1. 랜딩페이지 (구직자용)

### 목적
로그인한 구직자가 첫 화면에서 "내가 지금 어디에 합격 가능성이 있는지"를 한눈에 파악하게 한다.

### 핵심 사용자 스토리
- 구직자로서, 여러 기업 공고를 카드/대시보드 형태로 보면서 각 공고의 합격 확률을 바로 확인하고 싶다.
- 구직자로서, 이미 지원한 공고와 찜한 공고를 구분해서 보고 싶다.

### 주요 구성 요소
| 요소 | 설명 | 데이터 출처 |
|---|---|---|
| 공고 카드 리스트 | 회사 로고, 회사명, 직군, 마감일 | 원티드 `/jobs` |
| 합격 확률 배지 | 카드마다 %로 표시 | ⚠️ 유료 API 또는 자체 매칭 스코어 (섹션 1-1 참고) |
| 지원현황 태그 | "지원함 / 미지원" 표시 | Supabase `applications` 테이블 |
| 찜 아이콘 | 하트 토글 | Supabase `bookmarks` 테이블 |
| 필터/정렬 | 직군, 마감임박순, 합격확률순 | 클라이언트 사이드 필터링 |

### 1-1. 합격확률 산출 방식 — 반드시 결정해야 할 분기점
- **A안 (유료 계약)**: 원티드 AI 예측 API 사용. 정확도는 높지만 계약·비용 필요, 비동기 콜백 구조라 카드 목록 전체를 실시간 계산하지 못하고 백엔드에서 사전 배치 계산 후 캐싱해야 함.
- **B안 (자체 매칭 스코어, MVP 권장)**: 사용자의 스킬·자격증·경력 연차와 공고의 `skill_tags`/`requirements`/`annual_from~to`를 비교해 자체 스코어(0~100)를 산출. 정확도는 낮지만 계약 없이 즉시 구현 가능.
- **MVP는 B안으로 시작하고, 화면에는 "AI 예측 합격률(참고용)"이라고 명시해 실제 합격률처럼 오인되지 않게 한다.**

### 예외/엣지 케이스
- 비로그인 상태로 접근 시 → 합격확률/지원현황/찜 없이 공고 리스트만 노출, 로그인 유도 배너 표시
- 이력서/경력 정보를 아직 입력하지 않은 회원 → 합격확률 배지 대신 "마이페이지에서 경력을 입력하면 합격확률을 볼 수 있어요" 안내

## 2. 랜딩페이지 (기업용)

### 목적
기업 담당자가 자사에 자기소개서 정보를 제공한 구직자들을 인재상 부합도 순으로 확인한다.

### 주요 구성 요소
| 요소 | 설명 | 데이터 출처 |
|---|---|---|
| 지원자 카드 리스트 | 익명화된 프로필 요약 | Supabase `resumes`/`self_intros` (구직자가 정보 제공에 동의한 경우만) |
| 인재상 부합도 배지 | 회원가입 시 기업이 입력한 인재상과 비교한 점수 | 자체 매칭 로직 |
| 지원 여부 표시 | 이 구직자가 우리 회사에 실제 지원했는지 | Supabase `applications` (해당 기업 job_id로 필터) |

### ⚠️ 미해결 사항 (문서화만 하고 결정은 보류)
사용자가 지적한 대로, 기업이 인재를 "확인"만 할 수 있고 스카웃/우대 전형 연결이 불가능하면 기업 측 참여 유인이 약함. 이 문서에서는 다음 방향성 후보만 남겨두고, 팀 논의 후 확정한다.
- 후보 1: 기업이 관심 지원자에게 "면접 제안 메시지"를 보낼 수 있는 기능 추가 (스카웃은 아니지만 접촉 채널 제공)
- 후보 2: 인재상 부합도 리포트 자체를 유료 구독 상품화 (기업 대상 B2B 과금)
- 후보 3: MVP에서는 수익모델을 보류하고, "채용 시장 인사이트 제공"만으로 초기 트래픽 확보 후 후속 결정

## 3. 로그인/회원가입 페이지 (구직자용)

### 목적
Supabase Auth 기반으로 계정을 만들고, 이후 마이페이지/합격확률 계산에 쓰일 기본 정보를 최초 1회 수집한다.

### 입력 항목
아이디, 비밀번호, 학적(학교/전공/졸업여부), 학점, 자격증(복수), 경력(회사/기간/직무, 복수)

### 주요 흐름
1. 이메일/비밀번호로 Supabase Auth 가입
2. 가입 직후 온보딩 폼으로 학적·학점·자격증·경력 입력 (선택적으로 나중에 완성 가능하게 "건너뛰기" 허용)
3. 입력 완료 시 마이페이지로 이동

### 데이터 검증 규칙 (Backend-Guideline.md에 반영 필요)
- 학점: 4.5 만점/4.3 만점 등 학교별 스케일이 다를 수 있음 → 스케일 선택 필드 별도 필요
- 자격증/경력은 배열 형태로 여러 개 추가 가능해야 함 (동적 폼 필드)

### 예외 케이스
- 소셜 로그인 없이 이메일 기반만 MVP 범위로 한정 (추후 확장 여지로 남김)

## 4. 로그인/회원가입 페이지 (기업용)

### 목적
같은 가입 폼에서 "구직자/기업"을 선택 가능하게 하고, 기업 선택 시 인재상 정보를 수집한다.

### 입력 항목
아이디, 비밀번호, 회사명, 원하는 인재상(학점 기준, 기술스택, 유사 경험, 인턴십 경험 유무 등 구조화된 항목)

### MVP 범위의 명시적 생략 사항
- 기업 실체 인증(사업자등록증 검증 등) 절차는 생략. 이 부분은 화면에 "MVP 단계로 별도 인증 없이 자체 신고 기반으로 운영됩니다" 같은 문구를 넣어, 이후 실사용자 확장 시 반드시 인증 절차 추가가 필요함을 남겨둔다.

### 데이터 모델 초안
```
company_profile: user_id, company_name, preferred_gpa_min,
  preferred_skills[], preferred_experience_type[], internship_required(boolean)
```

## 5. 마이페이지 (구직자용)

### 목적
회원가입 시 입력한 경력·자격증 정보를 확인/수정하고, 지원현황·찜 목록을 모아본다.

### 주요 구성 요소
| 섹션 | 내용 | 데이터 출처 |
|---|---|---|
| 프로필 요약 | 학적, 학점, 자격증, 경력 리스트 | Supabase (섹션 3에서 입력한 값) |
| 정보 수정 | 회원가입 폼과 동일한 필드 재사용 | Supabase update |
| 나의 지원현황 | 지원한 공고 리스트 (자진신고 방식) | Supabase `applications` + 원티드 `/jobs/{id}` 재조회로 최신 상태 병합 |
| 찜 목록 | 찜한 공고 리스트 | Supabase `bookmarks` + `/jobs/{id}` 재조회 |

### 지원현황 기록 방식 (중요 — 다른 화면과 연결됨)
2번 페이지(기업별 상세정보)에서 "지원하기" 클릭 → 외부 이동 → 사용자가 앱으로 복귀 시 "지원을 완료하셨나요?" 확인 모달 → Yes 선택 시에만 `applications`에 기록. 이 로직은 마이페이지가 아니라 상세정보 페이지 담당이지만, 마이페이지는 그 결과를 읽기만 한다는 점을 명확히 한다.

### 목적
기업이 자사가 입력한 인재상 정보를 확인/수정하고, 지금까지 확인한 지원자 이력을 모아본다.

## 7. 기업별 상세정보 페이지

### 목적
구직자가 특정 공고를 자세히 보고, 실제 지원 여부를 결정하는 화면.

### 주요 구성 요소
| 요소 | 설명 | 데이터 출처 |
|---|---|---|
| 회사 소개 | 로고, 소개 이미지, 설명 | 원티드 `/companies/{id}` |
| 사내 복지/문화 태그 | company_tags 기반 | 원티드 `/companies/{id}` |
| 모집기간 | 마감일 | 원티드 `/jobs/{id}` (`due_time`) |
| 회사 평균연봉 (선택 표시) | 사업자번호 확보 시 | 원티드 `/insight/company` |
| **지원하기 버튼** | 클릭 시 원티드 실제 지원 페이지로 새 탭 이동 | `job.url` 필드 |
| 지원자 통계 | ❌ 미제공 (섹션 0 제약 참고) — 대체 지표로 "회사 최근 채용 활발도" 정도만 검토 |

### 지원 완료 판정 플로우 (섹션 5와 직결)
1. "원티드에서 지원하기 ↗" 버튼 클릭 → `window.open(job.url)`
2. 사용자가 탭 복귀 시(`visibilitychange` 이벤트) → "지원을 완료하셨나요?" 모달
3. Yes → Supabase `applications`에 기록 / No 또는 무응답 → 기록하지 않음

### UX 주의사항
버튼 라벨과 이동 시점에 "외부 사이트로 이동함"을 명확히 표시해, 사용자가 우리 앱에서 지원이 완결된다고 오인하지 않게 한다 (실제 제출 검증이 불가능하다는 기술적 한계에서 나온 필수 요구사항).

## 8. 지원서 작성 후 피드백 페이지

### 목적
구직자가 자기소개서를 입력하면, 목표 공고 대비 강약점과 경쟁자 비교 피드백을 제공한다. 이 데이터는 기업의 인재상 매칭(섹션 2)에도 함께 활용된다.

### 입력
- 자기소개서 텍스트
- 목표 공고 선택 (원티드 `/jobs/{id}` 중 하나)

### 출력 (LLM 기반 분석 — 원티드 API가 제공하지 않는 영역)
| 항목 | 산출 방식 |
|---|---|
| 피드백 | 자소서 + 공고(`intro`, `main_tasks`, `requirements`)를 LLM에 함께 넣어 "부합하는 점 / 보완할 점"을 생성 |
| 나의 장단점 | 이력서(경력·자격증) + 자소서를 LLM으로 구조화 분석 |
| 경쟁자 비교분석 | 같은 공고에 자소서를 제출한 다른 구직자들(익명화된 요약 통계)과 비교. **주의: 실제 경쟁자 수가 적으면 비교 자체가 무의미하거나 개인식별 위험이 있으므로, 최소 표본 수(예: 5명 이상) 미만이면 "비교 데이터 부족" 안내로 대체해야 함**

### 데이터 흐름과 기업 측 연결
사용자가 이 페이지에서 제출을 동의하면, 해당 자소서 요약이 섹션 2(기업용 랜딩페이지)의 인재상 매칭 대상에 포함된다. **동의 체크박스가 반드시 명시적으로 있어야 한다** (개인정보 제공 동의 없이 기업에 노출되면 안 됨).

### 데이터 모델 초안
```
self_intros: user_id, job_id, content, submitted_at,
  shared_with_company(boolean)  -- 기업 노출 동의 여부
feedback_results: self_intro_id, strengths[], improvements[], generated_at
```

## 10. 다음 단계 (팀 논의 필요)

1. **랜딩페이지(기업용)의 수익 구조** — 섹션 2의 후보 1/2/3 중 방향 결정 필요
2. **초기 이용자 확보 전략** — 이 문서 범위 밖이지만, 구직자 쪽 콘텐츠(합격확률 진단)를 무료 미끼로 먼저 트래픽을 모으고, 기업 쪽은 그 이후 순차 온보딩하는 순서를 제안. 구체 전략 문서는 별도 작성 필요
3. **합격확률 산출 방식(A안/B안) 최종 결정** — 이 문서는 B안(자체 매칭)을 MVP 기본값으로 가정하고 작성됨. A안으로 갈 경우 랜딩페이지(구직자용) 데이터 흐름 재설계 필요
