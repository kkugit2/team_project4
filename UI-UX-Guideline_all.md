## 0. 디자인 컨셉

### 왜 이 방향인가
이 서비스의 핵심 차별점은 "합격확률"이라는 모호한 숫자가 아니라, **"기업이 원하는 조건 중 몇 개를 충족했는가"라는 셀 수 있는 사실**이다. 그래서 매칭 게이지는 매끈한 퍼센트를 전면에 내세우지 않는다. 중앙에는 "3/4"처럼 **몇 개 중 몇 개를 충족했는지 절대 수치를 먼저** 보여주고, 퍼센트(75%)는 그 아래 보조 텍스트로만 작게 표기한다. "몇 개 중 몇 개"라는 사실이 먼저 읽혀야 하며, 퍼센트가 주가 되어서는 안 된다. 이게 이 프로덕트에서 가장 자주 보게 될 화면 요소이자 유일한 시그니처다. 나머지 UI는 이 시그니처가 튀도록 절제한다.

### 톤앤매너
전문적이되 딱딱하지 않게. 구직자에게는 "진단받는 느낌"이 아니라 "같이 보완점을 찾아가는 느낌"을, 기업에게는 "데이터 기반으로 신뢰할 수 있는 화면"이라는 인상을 준다.

## 1. 디자인 토큰

### 1-1. 컬러

| 이름 | HEX | 용도 |
|---|---|---|
| `--bg` | #F5F6F8 | 페이지 배경 |
| `--paper` | #FFFFFF | 카드/패널/폼 배경 |
| `--ink` | #131A24 | 기본 텍스트, 상단 유틸 요소 |
| `--muted` | #5B6472 | 보조 텍스트, 설명문, 타임스탬프 |
| `--line` | #E3E6EA | 보더, 구분선 |
| `--navy` (Primary) | #1D3557 | 버튼, 활성 탭, 게이지, 랭킹 배지 — **유일한 포인트 컬러** |
| `--navy-dark` | #122A46 | 버튼 hover, 강조 텍스트 |
| `--navy-100` | #E7ECF3 | 태그/칩 배경, 아이콘 배경 |
| `--navy-mid` | #4A6C93 | 보조 데이터 시각화(비교 기준선, 2번째 도넛 조각) |
| `--good` | #3E8F6F | 강점 카드 전용 (의미 색상, 포인트 컬러 아님) |
| `--warn` | #B4503B | 약점/경고 카드 전용, 찜(하트) 활성색 |

> **원칙**: 포인트 컬러는 남색(`--navy`) 하나로 통일한다. 강점(`--good`)/약점(`--warn`)은 브랜드 컬러가 아니라 의미 전달용 보조 색상이므로, 버튼이나 내비게이션 등 UI 골격에는 사용하지 않는다.

### 1-2. 타이포그래피

- 본문/UI: **Pretendard**
- 데이터/수치(통계 카드 수치, 매칭 게이지 숫자, 랭킹 번호, D-day, 학점): **JetBrains Mono** — "측정된 값"이라는 인상 부여
- 제목(`h1~h2`): Pretendard Bold, 본문 대비 크기 차이를 명확히 (18~24px)

### 1-3. 형태

- 카드/패널: `border-radius: 14px`, `1px solid var(--line)`, 그림자 없이 보더로만 구분 (플랫하고 묵직한 톤 유지)
- 버튼: `border-radius: 8~9px`
- 배지/칩/역할태그: `border-radius: 999px` 또는 `6px` (완전한 알약형은 상태 표시, 각진 소배지는 라벨 표시로 구분)

## 2. 시그니처 컴포넌트 — 매칭 게이지 (Circular Gauge)

### 형태
- 원형 프로그레스(SVG arc) + 중앙 표기 영역으로 구성한다.

### 중앙 표기 우선순위 (중요)
- 게이지 중앙에는 **"3/4" 형태의 절대 충족 개수를 JetBrains Mono체로 크게** 표기한다.
- 퍼센트(예: 75%)는 그 절대 수치 **아래에 보조 텍스트로만 작게** 표기한다.
- 이는 "이 서비스가 다루는 것은 감성적 평가가 아니라 셀 수 있는 사실"이라는 톤을 시각적으로 계속 상기시키기 위함이며, "몇 개 중 몇 개를 충족했는가"라는 사실이 퍼센트보다 먼저 읽혀야 한다.

### 색상 규칙
- 70점 이상: `--navy`
- 40~69점: `--navy-mid`
- 40점 미만: `--warn`

### 사용처
- 구직자 공고 카드의 합격확률, 기업 인재 랭킹의 부합도 등 **매칭도가 나오는 모든 화면에서 동일 컴포넌트를 재사용**한다.
- 카드 목록(공고 카드, 랭킹 리스트)에서는 소형 사이즈, 상세 화면(기업 상세정보)에서는 대형 사이즈 + 항목별 체크리스트를 함께 노출한다.

### 예외 상태
- 조건이 등록되지 않은 기업이거나 구직자가 경력 정보를 미입력한 경우, 게이지를 그리지 않고 안내 문구로 대체한다 (배지/게이지 자리를 비워두지 않는다).

## 3. 레이아웃 원칙

- 데스크톱 고정폭 기준(`min-width: 1280px` 가정), 본문 컨텐츠 최대폭 1160px, 중앙 정렬
- 모바일 반응형은 이번 범위 밖으로 한다. 단, 그리드 컬럼 수 변경 시에는 최소 너비(`minmax`)를 지정해 카드가 과도하게 좁아지지 않게 한다.
- **상단 메뉴바(top nav)**: 모든 로그인 후 화면에 고정. 좌측 로고, 좌측 상단 메뉴 링크(현재 화면 강조는 `border-bottom: 2px solid var(--navy)`), 우측 역할 태그 + 아바타
- 역할(구직자/기업)에 따라 메뉴 구성이 다르다:
  - 구직자: 홈 / 공고 둘러보기 / 마이페이지
  - 기업: 홈 / 스카웃 관리 / 마이페이지
- 폼(입력) + 결과(출력) 조합 화면은 좌:우 = 260~360px : 1fr 비율의 2컬럼

## 4. 화면별 가이드

### 4-1. 회원가입 (역할 우선 분기 — 중요)
- **1단계**: "구직자로 시작하기" / "기업으로 시작하기" 두 개의 큰 선택 카드만 노출한다. 로그인 폼 요소는 아직 보이지 않는다.
- **2단계**: 선택한 역할의 입력 폼만 표시한다. 반대 역할의 필드는 **화면에 아예 존재하지 않아야 한다** (토글로 숨기는 방식 금지 — 코드상 조건부 렌더링으로 분리).
- 상단에 "역할 다시 선택" 링크로만 되돌아갈 수 있게 한다.
- 가입 완료 시 각 역할에 맞는 홈으로 자동 이동 (구직자 → 통계 홈 / 기업 → 랭킹 홈).

### 4-2. 구직자 홈 (로그인 후 첫 화면 = 통계 분석)
- 상단 통계 카드 4개(연차/스킬 수/자격증/지원 가능 공고 수) → 비교 바 차트 → 스킬 구성 도넛 → 강점/약점 카드 순서로 배치
- 이 화면은 공고 리스트가 아니라 **본인 데이터에 대한 분석 결과**라는 점을 명확히 한다 (공고 탐색은 별도 "공고 둘러보기" 메뉴로 분리)

### 4-3. 기업 홈 (로그인 후 첫 화면 = 인재 랭킹)
- 등록된 채용 조건 요약(칩) + "채용 조건 수정" 링크를 상단에 고정
- 순위 번호(mono) + 매칭 게이지 + 스킬 칩 + 스카웃 버튼을 한 행에 배치
- 스카웃 버튼은 상태에 따라 라벨 변경: `스카웃 제안` → `스카웃 발송됨 · 대기중` (전송 후 비활성 톤으로 전환)

### 4-4. 스카웃 관리 (기업 전용)
- 마이페이지의 요약과는 별개로, 보낸 스카웃만 모아 상태별(전체/대기중/수락됨/거절됨/만료됨) 필터로 관리하는 전용 화면
- 각 행에 발송일, 남은 만료 기한, 상태별 액션 버튼(대기중 → 제안 취소, 수락됨 → 지원 페이지로) 노출
- 이번 달 발송 건수/한도(예: 7 / 10건)를 화면 상단에 항상 표시해 발송 제한을 인지시킨다
- 마이페이지 쪽 요약 리스트에는 "전체보기 →" 링크로 이 화면과 연결

### 4-5. 공고 둘러보기 (구직자)
- 필터 pill(직군별) + 정렬 드롭다운을 카드 그리드 위에 배치
- 카드 클릭 시 기업 상세정보로 이동
- 경력 미입력 회원은 매칭 게이지 대신 안내 문구로 대체 (게이지 자리 비워두지 않기)

### 4-6. 기업 상세정보
- 상단: 회사 소개(이미지, 설명) → 중단: 매칭 게이지(대형) + 항목별 체크리스트("학점 ✓ / 기술스택 ✓ / 유사경험 ✗ / 인턴십 ✓") → 하단: 지원하기 버튼
- **지원 버튼 문구는 반드시 "원티드에서 지원하기 ↗"로 통일한다** (화살표 아이콘 필수 — 외부 이동임을 아이콘으로도 이중 표시)
- 복귀 시 뜨는 완료 확인 모달 문구:
  ```
  제목: 지원을 완료하셨나요?
  본문: 원티드 페이지에서 지원서 제출까지 마치셨다면 알려주세요.
  버튼: [아직요] [네, 지원했어요]
  ```
  "아직요"를 누르거나 모달을 닫아도 페널티 없이 다시 물어볼 수 있게, 지원 기록을 강제하지 않는다. 실제 구현 시 `visibilitychange` 이벤트 기반으로만 등장하게 한다.

### 4-7. 자소서 피드백
- 자소서 원문(읽기전용) → 강점(연한 초록 배경)/약점(연한 브릭레드 배경) 2단 카드 → 경쟁자 비교(표본 5명 미만 시 아래 문구로 대체) → 기업 공유 동의 체크박스 순서
- 경쟁자 비교 섹션이 표본 부족(5건 미만)일 때 문구:
  ```
  아직 이 공고에 제출된 자소서가 충분하지 않아요. 비교 데이터가 쌓이면 보여드릴게요.
  ```
  빈 그래프나 회색 처리된 가짜 차트를 보여주지 않는다 (있는 것처럼 보이는 로딩 상태 금지).
- 동의 체크박스는 반드시 명시적으로 노출하고, **기본값은 비동의(미체크)로 시작한다** (옵트인 원칙, 실수로 공유되지 않게).
- 체크박스 옆 보조 설명: "동의하시면 인재상이 맞는 기업이 익명화된 요약으로 확인할 수 있어요"

### 4-8. 마이페이지 (구직자 / 기업)
- 좌: 프로필/인재상 요약 카드(수정 버튼 하단 고정) · 우: 목록형 섹션들(지원현황/찜/스카웃함 또는 열람로그/스카웃 발송내역)
- 목록 섹션이 많아질 경우 "전체보기" 링크로 전용 관리 화면(예: 스카웃 관리)으로 분리한다 — 마이페이지 자체는 요약만 담당한다.

## 5. 톤 & 라이팅

- 사용자가 통제하는 것을 기준으로 이름 짓기 (예: "매칭 재계산"이 아니라 "다시 확인하기")
- 실패/빈 상태는 원인과 다음 행동을 함께 제시 (예: "비교 데이터 부족 — 제출 인원이 5명 미만입니다")
- 버튼 라벨과 그 결과로 뜨는 안내 문구의 단어를 일치시킨다 (예: "지원하기" 버튼 → "지원을 완료하셨나요?" 모달)
- 존댓말 유지, 과도한 감탄사·이모지 지양
- 외부 이동, 데이터 미제공, 계약 필요 등 기술적 한계가 있는 지점은 반드시 화면 문구로 명시한다 (예: "AI 예측 합격률(참고용)", "익명화됨")

### 고정 문구 재사용 원칙 (중요)
아래 문구는 화면마다 임의로 바꿔 쓰지 않고 **정확히 동일한 문구를 재사용**한다 — 반복되는 문구의 일관성이 곧 사용자가 서비스를 신뢰하는 방식이다.

| 항목 | 고정 문구 |
|---|---|
| 지원 버튼 (기업 상세정보) | 원티드에서 지원하기 ↗ |
| 완료 확인 모달 제목 | 지원을 완료하셨나요? |
| 완료 확인 모달 버튼 | [아직요] [네, 지원했어요] |
| 표본 부족 안내 (자소서 피드백) | 아직 이 공고에 제출된 자소서가 충분하지 않아요. 비교 데이터가 쌓이면 보여드릴게요. |
| 자소서 공유 동의 보조설명 | 동의하시면 인재상이 맞는 기업이 익명화된 요약으로 확인할 수 있어요 |

## 6. 접근성 · 품질 기준

- 모든 인터랙티브 요소(버튼, 링크, 체크박스)에 시각적 포커스 표시를 항상 유지한다 (커스텀 스타일링으로 제거 금지).
- 상태를 색상에만 의존해 전달하지 않는다 — 매칭 게이지/배지는 색상 + 숫자(또는 텍스트 라벨)를 항상 함께 표기한다.
- 데스크톱 전제이므로 반응형 그리드는 필수는 아니지만, 그리드 컬럼 수 변경 시 최소 너비(`minmax`)를 지정해 카드가 과도하게 좁아지지 않게 한다.
- WCAG 2.1 AA 기준(명도 대비 4.5:1 이상)을 준수한다.

## 7. 공통 상태 처리 원칙

| 상태 | 원칙 |
|---|---|
| 로딩 | 스켈레톤 UI 사용, 스피너 남발 금지 |
| 에러 | "무엇이 잘못됐고 어떻게 해야 하는지"를 반드시 함께 안내. "오류가 발생했습니다" 단독 문구 금지 |
| 빈 화면 | 행동을 유도하는 문구로 작성. 예: 찜 목록이 비었을 때 → "관심 있는 공고에 하트를 눌러보세요" |
| 외부 API 실패 | 캐시 폴백과 연동 — "최신 정보가 아닐 수 있어요" 배지를 캐시 데이터 위에 작게 표기 |

**문서 유지보수**
- 모든 색상, 폰트, 간격은 CSS 변수로 관리한다.
- 5장의 고정 문구는 임의로 수정하지 않으며, 변경이 필요하면 이 문서를 먼저 개정한 뒤 화면에 반영한다.
- 이 문서는 3종 초안(디자인 상세 스펙안 / 게이지·데스크톱 중심안 / 세그먼트·모바일 대응안)을 검토해 확정한 통합본이다.
