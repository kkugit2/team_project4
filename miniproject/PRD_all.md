# PRD.md — 채용 정보 매칭 플랫폼 (통합본)

> 기업 특강 실습 프로젝트 · 데모(MVP) 버전
> 통합 대상: 팀 공통 PRD 초안 + 담당 페이지 상세 명세(PRD (2).md)
> 최종 갱신: 2026-07-14

---

## 0. 서비스 한 줄 요약

원티드 오픈 API의 실제 채용공고 데이터를 기반으로, 구직자에게는 **"합격 가능성 진단 + 자기계발 방향 제시"**를,
기업에게는 **"인재상에 부합하는 지원자 발굴 + 스카웃 채널"**을 제공하는 양방향 매칭 플랫폼.

## 1. 왜 만드는가 (문제 정의)
- 구직자는 자신의 이력이 시장에서 어느 정도 위치인지, 무엇을 보완해야 하는지 객관적으로 알기 어렵다.
- 기업은 방대한 지원자 중 자사 인재상에 맞는 사람을 찾는 데 시간이 많이 든다.
- 원티드 API는 실제 채용공고/기업 데이터를 제공하지만, "분석·추천·매칭 근거"까지는 제공하지 않는다. → 이 서비스가 그 간극을 채운다.

## 2. 대상 사용자
- **구직자**: 신입~주니어 개발/데이터 직군, 자신의 시장 위치를 파악하고 싶은 사람
- **기업 인사담당자**: 채용 중인 포지션에 맞는 인재를 사전에 탐색하고 싶은 담당자

## 3. 핵심 제약 (모든 화면 설계의 전제)
원티드 API의 실제 스펙에서 확인된 제약으로, 화면 설계 전체에 영향을 준다.

- 원티드 API는 **조회 전용**이다. 실제 지원서 제출 API가 없으므로 "지원하기"는 항상 **외부 사이트로의 이동**이다.
- 연봉·경쟁률·지원자 수는 공개 API에 없다. 있는 것은 회사 단위 평균연봉(`/insight/company`, 사업자번호 필요)뿐이다.
- 합격확률은 원티드 유료 AI 예측 API(`/ai/pass/text-prediction`)를 쓰거나, 자체 매칭 로직(스킬/자격증/경력 비교)으로 대체한다.
  - **A안(유료 계약)**: 정확도는 높지만 계약·비용 필요, 비동기 콜백 구조라 카드 목록 전체를 실시간 계산 불가 → 백엔드 사전 배치 계산 후 캐싱 필요
  - **B안(자체 매칭 스코어)**: 스킬·자격증·경력 연차와 공고의 `skill_tags`/`requirements` 비교로 0~100점 자체 산출. 계약 없이 즉시 구현 가능
  - **→ MVP는 B안으로 시작**하고, 화면에는 "AI 예측 합격률(참고용)"이라고 명시해 실제 합격률로 오인되지 않게 한다.
- 회원 데이터(경력, 자격증, 지원현황, 찜 목록, 자소서)는 전부 Supabase에 직접 저장·관리한다.

## 4. 화면별 상세 명세

### 4-1. 랜딩페이지 (구직자용)
**목적**: 로그인한 구직자가 첫 화면에서 "지금 어디에 합격 가능성이 있는지" 한눈에 파악

| 요소 | 설명 | 데이터 출처 |
|---|---|---|
| 공고 카드 리스트 | 회사 로고, 회사명, 직군, 마감일 | 원티드 `/jobs` |
| 합격 확률 배지 | 카드마다 %로 표시 (B안 기준) | 자체 매칭 스코어 |
| 지원현황 태그 | "지원함 / 미지원" | Supabase `applications` |
| 찜 아이콘 | 하트 토글 | Supabase `bookmarks` |
| 필터/정렬 | 직군, 마감임박순, 합격확률순 | 클라이언트 사이드 |

**예외 케이스**
- 비로그인 접근 → 합격확률/지원현황/찜 없이 공고 리스트만, 로그인 유도 배너
- 경력 정보 미입력 회원 → 배지 대신 "마이페이지에서 경력을 입력하면 합격확률을 볼 수 있어요" 안내

### 4-2. 랜딩페이지 (기업용)
**목적**: 기업 담당자가 정보 제공에 동의한 구직자를 인재상 부합도 순으로 확인

| 요소 | 설명 | 데이터 출처 |
|---|---|---|
| 지원자 카드 리스트 | 익명화된 프로필 요약 (정보 제공 동의자만) | Supabase `self_intros` (`shared_with_company=true`) |
| 인재상 부합도 배지 | 기업이 입력한 인재상과 비교한 점수 | 자체 매칭 로직 |
| 지원 여부 표시 | 해당 구직자의 자사 지원 여부 | Supabase `applications` (job_id 필터) |
| 스카웃 버튼 | 관심 인재에게 제안 발송 | Supabase `scouts` (4-2-1 참고) |

**4-2-1. 스카웃 장치 (기업 실익 구조 — 확정)**
> 이전 논의에서 "기업이 확인만 가능하고 실질적 컨택 수단이 없다"는 문제가 제기되었고, 스카웃 기능을 추가하는 것으로 결정됨.
- 스카웃 제안 시 **기업명·포지션 공개**, 연봉/조건은 선택 공개
- 구직자 반응: **수락 / 거절 / 무응답**(7일 후 자동 만료)
- 기업당 **월 발송 건수 제한**(기본값 10건, 서버단 강제) — 스팸 방지
- 스카웃 = "관심 표시"이며 프리패스 아님. 수락 시 정식 지원 페이지로 연결, 기업 쪽엔 "스카웃 수락자"로 별도 표시되어 우선 검토 유도

### 4-3. 로그인/회원가입 (구직자용)
**입력 항목**: 아이디, 비밀번호, 학적(학교/전공/졸업여부), 학점(+ 스케일 선택, 예: 4.5/4.3만점), 자격증(복수, 동적 필드), 경력(회사/기간/직무, 복수)

**흐름**: 이메일/비밀번호 가입 → 온보딩 폼(건너뛰기 허용) → 마이페이지 이동

**MVP 범위**: 소셜 로그인 제외, 이메일 기반만 지원

### 4-4. 로그인/회원가입 (기업용)
**입력 항목**: 아이디, 비밀번호, 회사명, 원하는 인재상(학점 기준, 기술스택, 유사 경험, 인턴십 경험 유무)

**명시적 생략**: 기업 실체 인증(사업자등록증 검증 등) 생략. 화면에 "MVP 단계로 별도 인증 없이 자체 신고 기반으로 운영됩니다" 안내 문구 필요 (추후 확장 시 반드시 추가할 항목으로 기록)

```
company_profile: user_id, company_name, preferred_gpa_min,
  preferred_skills[], preferred_experience_type[], internship_required(boolean)
```

### 4-5. 마이페이지 (구직자용)
| 섹션 | 내용 | 데이터 출처 |
|---|---|---|
| 프로필 요약 | 학적, 학점, 자격증, 경력 | Supabase (회원가입 값) |
| 정보 수정 | 회원가입 폼 필드 재사용 | Supabase update |
| 나의 지원현황 | 지원한 공고 리스트 (자진신고) | `applications` + 원티드 `/jobs/{id}` 병합 |
| 찜 목록 | 찜한 공고 리스트 | `bookmarks` + `/jobs/{id}` 병합 |
| 스카웃함 | 받은 스카웃 제안 목록, 수락/거절 처리 | `scouts` |

### 4-6. 마이페이지 (기업용)
| 섹션 | 내용 |
|---|---|
| 인재상 프로필 | 회원가입 시 입력 기준 (수정 가능) |
| 확인한 지원자 로그 | 랜딩페이지(기업용)에서 열람한 지원자 히스토리 |
| 스카웃 발송 내역 | 보낸 스카웃 상태(수락/거절/만료) 추적 |

### 4-7. 기업별 상세정보 페이지
| 요소 | 설명 | 데이터 출처 |
|---|---|---|
| 회사 소개 | 로고, 소개 이미지, 설명, 복지/문화 태그 | 원티드 `/companies/{id}` |
| 모집기간 | 마감일(`due_time`) | 원티드 `/jobs/{id}` |
| 회사 평균연봉(선택 표시) | 사업자번호 확보 시 | 원티드 `/insight/company` |
| **지원하기 버튼** | 클릭 시 원티드 실제 지원 페이지로 새 탭 이동 | `job.url` |
| 지원자 통계 | ❌ 미제공 (섹션 3 제약) | — |

**지원 완료 판정 플로우** (마이페이지 지원현황과 직결)
1. "원티드에서 지원하기 ↗" 클릭 → `window.open(job.url)`
2. 탭 복귀 시(`visibilitychange`) → "지원을 완료하셨나요?" 확인 모달
3. Yes → `applications`에 기록(`status: self_reported`) / No·무응답 → 기록 안 함

**UX 주의사항**: 버튼 라벨과 이동 시점에 "외부 사이트로 이동함"을 명확히 표시 — 실제 제출 여부를 기술적으로 검증할 수 없다는 한계에서 나온 필수 요구사항

### 4-8. 지원서 작성 후 피드백 페이지
**입력**: 자기소개서 텍스트, 목표 공고 선택

| 항목 | 산출 방식 |
|---|---|
| 피드백 | 자소서 + 공고(`intro`,`main_tasks`,`requirements`)를 LLM에 함께 입력해 "부합하는 점/보완할 점" 생성 |
| 나의 장단점 | 이력서(경력·자격증) + 자소서를 LLM으로 구조화 분석 |
| 경쟁자 비교분석 | 같은 공고에 제출한 다른 구직자(익명화 요약 통계)와 비교. **표본 5명 미만이면 "비교 데이터 부족" 안내로 대체** (무의미한 비교·개인식별 위험 방지) |

**동의 플로우 (중요)**: 이 페이지에서 제출 동의 시에만 해당 자소서 요약이 4-2(기업용 랜딩페이지) 매칭 대상에 포함됨. **명시적 동의 체크박스 필수** (동의 없이 기업에 노출 금지)

## 5. 통합 데이터 모델 (Supabase)

```
profiles (Auth 확장)
  - id, role: 'jobseeker' | 'company', created_at

jobseeker_profile
  - user_id, school, major, graduation_status, gpa, gpa_scale,
    certifications[], career_history(jsonb), resume_text

company_profile
  - user_id, company_name, wanted_company_id,
    preferred_gpa_min, preferred_skills[], preferred_experience_type[],
    internship_required(boolean)

applications
  - user_id, job_id, status('self_reported'), applied_at

bookmarks
  - user_id, job_id, created_at

self_intros
  - user_id, job_id, content, submitted_at, shared_with_company(boolean)

feedback_results
  - self_intro_id, strengths[], improvements[], generated_at

match_scores
  - jobseeker_id, company_id, score, basis(jsonb), computed_at

scouts
  - id, company_id, jobseeker_id, status('sent'|'accepted'|'rejected'|'expired'),
    sent_at, expires_at

job_cache (원티드 API 캐싱, rate limit 대응)
  - wanted_job_id, payload(jsonb), fetched_at
```

## 6. Out of Scope (MVP 범위 밖)
- 초기 이용자 확보 전략 → 홍보/마케팅 영역으로 분류, 본 PRD 범위 제외 (판단 완료)
- 기업 실명 인증 절차 (사업자등록증 검증 등)
- 유료 원티드 AI 예측 API 연동 (`/ai/pass`, `/ai/apply`) — 계약 필요, MVP는 자체 로직(B안)으로 대체
- 결제/구독 모델 (기업 인재상 리포트 유료화는 후보로만 남기고 보류)
- 소셜 로그인

## 7. 성공 기준 (데모 기준)
- 구직자 플로우(회원가입 → 자소서 분석 → 추천 확인 → 지원 기록) end-to-end 동작
- 기업 플로우(회원가입 → 인재상 등록 → 랭킹 확인 → 스카웃 발송 → 수락/거절 반영) end-to-end 동작
- 실제 원티드 API 데이터가 최소 1개 화면 이상에 반영

## 8. 결정된 사항 요약
| 이슈 | 결정 |
|---|---|
| 기업 실익 구조 | 스카웃 장치 도입으로 확정 (4-2-1) |
| 초기 이용자 확보 | 마케팅 영역, 본 PRD 범위 제외 |
| 합격확률 산출 | MVP는 B안(자체 매칭 스코어)으로 시작 |

## 9. 열린 질문 (다음 논의 필요)
- 매칭 점수 산출 로직의 구체적 가중치 (스킬/연차/학점 반영 비율)
- 자소서 분석에 사용할 LLM 프롬프트 표준화 (일관성 확보)
- 스카웃 수락률이 낮을 경우 기업 쪽 재참여 유도 방안
- 학점 스케일(4.5/4.3 등) 정규화 방식
- 경쟁자 비교분석 최소 표본(5명) 미만일 때의 대체 UI 문구 확정
- 기업 인재상 리포트 유료 구독 전환 여부 (보류 상태, 추후 재논의)
