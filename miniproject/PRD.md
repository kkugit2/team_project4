# PRD.md — 채용 정보 매칭 플랫폼 (MVP)

> 기업 특강 실습 프로젝트 · 데모(MVP) 버전
> 최종 갱신: 2026-07-14

## 1. 한 줄 정의
원티드 API의 실제 채용 데이터를 기반으로, 구직자에게는 "지금 상태로 지원 가능한 기업과 부족한 점"을,
기업에게는 "익명화된 인재 랭킹과 스카웃 채널"을 제공하는 양방향 매칭 서비스.

## 2. 왜 만드는가 (문제 정의)
- 구직자는 자신의 이력이 시장에서 어느 정도 위치인지, 무엇을 보완해야 하는지 객관적으로 알기 어렵다.
- 기업은 방대한 지원자 중 자사 인재상에 맞는 사람을 찾는 데 시간이 많이 든다.
- 원티드 API는 실제 채용공고/기업 데이터를 제공하지만, "분석·추천·매칭 근거"까지는 제공하지 않는다.
  → 이 서비스가 그 간극을 채운다.

## 3. 대상 사용자
- **구직자**: 신입~주니어 개발/데이터 직군, 자신의 시장 위치를 파악하고 싶은 사람
- **기업 인사담당자**: 채용 중인 포지션에 맞는 인재를 사전에 탐색하고 싶은 담당자

## 4. 핵심 기능 (구직자)
| # | 기능 | 설명 |
|---|---|---|
| 1 | 랜딩페이지 | 채용중 기업을 카드/대시보드 형태로 노출, 원티드 API 기반 합격 확률(자체 산출) 표시, 지원현황·찜 기능 |
| 2 | 기업 상세 | 회사 소개 확인, '지원하기' 클릭 시 실제 기업 지원 페이지로 이동 |
| 3 | 마이페이지 | 회원가입 시 입력한 학적/학점/자격증/경력 확인·수정 |
| 4 | 로그인/회원가입 | Supabase Auth 이용. 아이디, 비밀번호, 학적, 학점, 자격증, 경력 입력 |
| 5 | 자소서 분석 | 자소서 입력 → 장단점, 경쟁자 비교분석, 피드백 (기업이 등록한 인재상 데이터 참고) |
| 6 | What-if | 준비 중인 자격증 취득 시 매칭도 변화 시뮬레이션 |
| 7 | 모의면접 | AI 기반 예상 질문·답변 피드백 |

## 5. 핵심 기능 (기업)
| # | 기능 | 설명 |
|---|---|---|
| 1 | 로그인/회원가입 | 가입 시 기업/구직자 선택. 기업 인증은 MVP 단계에서 생략. 원하는 인재상 저장(학점, 기술스택, 유사 경험, 인턴십 여부 등) |
| 2 | 인재 대시보드 | 구직자가 제공한 자소서를 익명화하여 대시보드로 열람, 등록한 인재상과의 일치도(%) 확인, 자사 지원 여부 확인 |
| 3 | 스카웃 | 관심 있는 인재에게 스카웃 제안 발송 (아래 5-1 참고) |

### 5-1. 스카웃 장치 (기업 실익 구조 해결안)
- 스카웃 제안 시 **기업명·포지션 공개**, 연봉/조건은 선택 공개
- 구직자 반응: **수락 / 거절 / 무응답**(7일 후 자동 만료)
- 기업당 **월 발송 건수 제한**(기본값 10건) — 스팸 방지
- 스카웃은 "관심 표시"이며 프리패스 아님. 수락 시 정식 지원 페이지로 연결되되, 기업 쪽에는 "스카웃 수락자"로 별도 표시되어 우선 검토 유도

## 6. 범위 밖 (Out of Scope, MVP)
- 초기 이용자 확보 전략 → 홍보/마케팅 영역으로 분류, 본 PRD 범위 제외
- 기업 실명 인증 절차 (실제 사업자 확인 등)
- 유료 원티드 AI 예측 API(`/ai/pass`, `/ai/apply`) 연동 — 별도 계약 필요, MVP에서는 자체 임베딩+LLM 로직으로 대체
- 결제/구독 모델

## 7. 데이터 소스
- **실데이터**: 원티드 API (기업 정보, 채용공고, 직군/스킬 태그)
- **사용자 입력 데이터**: 회원가입 시 학적/학점/자격증/경력, 자소서 텍스트 (Supabase 저장)
- **파생 데이터**: 매칭 점수, 강점/약점 분석 결과 (서버에서 계산 후 저장 또는 캐싱)

## 8. 기술 스택 (요약, 상세는 Backend-Guideline.md / UI-UX-Guideline.md 참고)
- 프론트엔드: HTML / CSS / JavaScript
- 백엔드/DB/Auth: Supabase
- 배포: Vercel
- 외부 API: 원티드 OpenAPI

## 9. 성공 기준 (데모 기준)
- 구직자 플로우(회원가입 → 자소서 분석 → 추천 확인) end-to-end 동작
- 기업 플로우(회원가입 → 인재상 등록 → 랭킹 확인 → 스카웃 발송) end-to-end 동작
- 실제 원티드 API 데이터가 최소 1개 화면 이상에 반영

## 10. 열린 질문 (다음 논의 필요)
- 스카웃 수락률이 낮을 경우 기업 쪽 재참여를 어떻게 유도할지
- 매칭 점수 산출 로직의 구체적 가중치 (스킬/연차/학점 반영 비율)
- 자소서 분석에 사용할 LLM 프롬프트 표준화 (일관성 확보)
