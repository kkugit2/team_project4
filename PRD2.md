# PRD.md — 담당 페이지 명세서 (v2)

> 담당 범위: 랜딩페이지(구직자/기업), 로그인·회원가입 페이지(구직자/기업), 마이페이지(구직자/기업), 기업별 상세정보 페이지, 지원서 작성 후 피드백 페이지
> v2 변경사항: ① 원티드 API 실사용 가능 범위를 **회사·포지션·검색·통계·태그 5개 카테고리**로 확정 ② "합격확률"을 폐기하고 **"기업 요구사항 충족률(%)"**로 대체

---

## 0. 서비스 한 줄 요약

원티드 오픈 API의 실제 채용공고 데이터를 기반으로, 구직자에게는 "기업이 원하는 조건을 얼마나 충족하는지 진단 + 자기계발 방향 제시"를, 기업에게는 "인재상에 부합하는 지원자 발굴"을 제공하는 양면 플랫폼.

### 원티드 API 사용 가능 범위 (확정)
| 카테고리 | 용도 |
|---|---|
| 회사 (Company) | 회사 소개, 로고, 이미지, 복지 태그 조회 |
| 포지션 (Job) | 공고 목록/상세 조회 (마감일, 요구 연차, 스킬 태그 등) |
| 검색 (Search) | 회사명/포지션 검색, 자동완성 |
| 태그 (Tag) | 직군/스킬/매력태그 마스터 리스트 — 회원가입 폼의 스킬 선택 드롭다운 표준화에 사용 |
| 통계 (Stat) | 지원 현황 통계 — **단, 원티드에 등록된 실제 클라이언트(기업) 소유 공고에 한정**. 우리 플랫폼이 스크랩해오는 제3의 기업 공고에는 적용 불가하므로, 우리 자체 통계(Supabase 집계)로 대체 |

**이전 버전과 달라진 핵심 전제**
- AI 합격예측 API(`/ai/pass/text-prediction`)와 `recruit-company`/`ats`(기업 관리용 API)는 **사용 범위에서 완전히 제외**한다. 이번 버전에서는 이 두 카테고리를 참조하지 않는다.
- 그 결과 "합격확률"이라는 개념 자체를 없애고, **우리 자체 데이터(기업 회원가입 시 입력한 인재상)와 구직자 정보를 규칙 기반으로 비교한 "요구사항 충족률(%)"**로 전면 대체한다. 이는 외부 API 의존 없이 100% 자체 로직으로 계산 가능하다.

## 1. 랜딩페이지 (구직자용)

### 목적
로그인한 구직자가 첫 화면에서 "각 기업이 원하는 조건을 내가 얼마나 충족하는지"를 한눈에 파악하게 한다.

### 핵심 사용자 스토리
- 구직자로서, 여러 기업 공고를 카드/대시보드 형태로 보면서 각 공고에 대한 요구사항 충족률을 바로 확인하고 싶다.
- 구직자로서, 이미 지원한 공고와 찜한 공고를 구분해서 보고 싶다.

### 주요 구성 요소
| 요소 | 설명 | 데이터 출처 |
|---|---|---|
| 공고 카드 리스트 | 회사 로고, 회사명, 직군, 마감일 | 원티드 포지션(Job) API |
| **요구사항 충족률 배지** | "요구사항 4개 중 3개 충족 (75%)" 형태로 표시 | 자체 로직 (섹션 1-1) |
| 지원현황 태그 | "지원함 / 미지원" 표시 | Supabase `applications` |
| 찜 아이콘 | 하트 토글 | Supabase `bookmarks` |
| 필터/정렬 | 직군, 마감임박순, 충족률순 | 클라이언트 사이드 필터링 |

### 1-1. 요구사항 충족률 산출 방식 (확정 로직)

기업이 회원가입 시 입력한 인재상(`company_profile`)과 구직자의 `seeker_profile`을 1:1로 비교하는 **완전 규칙 기반 계산**이다. 외부 API나 AI 예측 없이 즉시 구현 가능.

```
충족 항목 수 = 0
전체 항목 수 = 기업이 설정한 조건 개수 (보통 4개: 학점, 기술스택, 유사경험, 인턴십 여부)

- 학점: seeker.gpa >= company.preferred_gpa_min → 충족
- 기술스택: seeker.skills 와 company.preferred_skills 교집합이 1개 이상 → 충족
  (※ 이때 스킬명 표기가 서로 달라 매칭 실패하는 걸 막기 위해,
     원티드 태그(Tag) API의 스킬 마스터 리스트를 두 폼 모두에서
     동일한 드롭다운/자동완성으로 쓰도록 강제한다)
- 유사경험: seeker.career_history 중 하나라도 company.preferred_experience_type과 일치 → 충족
- 인턴십 경험: company.internship_required가 true인데 seeker 경력에 인턴 항목이 있으면 충족
  (기업이 인턴십을 요구하지 않으면 이 항목은 계산에서 제외)

충족률(%) = (충족 항목 수 / 전체 항목 수) * 100
```

이 계산은 페이지 진입 시 클라이언트에서 즉시 실행 가능한 수준이라, 별도 배치 처리나 캐싱 없이 실시간 표시가 가능하다 (이전 버전의 "합격확률 캐싱 문제"가 이 방식으로 해소됨).

### 예외/엣지 케이스
- 비로그인 상태로 접근 시 → 충족률/지원현황/찜 없이 공고 리스트만 노출
- `seeker_profile`을 아직 완성하지 않은 회원 → 충족률 배지 대신 "마이페이지에서 정보를 입력하면 충족률을 볼 수 있어요" 안내
- 해당 기업이 인재상을 하나도 입력하지 않은 경우(전체 항목 수 = 0) → "이 기업은 아직 조건을 등록하지 않았어요" 표시하고 퍼센트 계산 자체를 생략

## 2. 랜딩페이지 (기업용)

### 목적
기업 담당자가 자사에 자기소개서 정보를 제공한 구직자들을 인재상 충족률 순으로 확인한다.

### 주요 구성 요소
| 요소 | 설명 | 데이터 출처 |
|---|---|---|
| 지원자 카드 리스트 | 익명화된 프로필 요약 | Supabase `resumes`/`self_intros` (구직자 동의 시에만) |
| **인재상 충족률 배지** | 1-1과 동일한 계산식을 기업 시점에서 표시 | 자체 로직 |
| 지원 여부 표시 | 이 구직자가 우리 회사에 실제 지원했는지 | Supabase `applications` |

### ⚠️ 미해결 사항 (결정 보류, 문서화만 유지)
기업이 인재를 "확인"만 가능하고 스카웃/우대 전형 연결이 안 되면 참여 유인이 약함. 방향성 후보 3가지는 유지한다.
- 후보 1: 관심 지원자에게 "면접 제안 메시지" 전송 기능
- 후보 2: 충족률 리포트 자체를 유료 구독 상품화 (B2B 과금)
- 후보 3: MVP는 수익모델 보류, 인사이트 제공만으로 초기 트래픽 확보 후 결정

## 3. 로그인/회원가입 페이지 (구직자용)

### 목적
Supabase Auth 기반 계정 생성 + 충족률 계산에 쓰일 기본 정보 수집.

### 입력 항목
아이디, 비밀번호, 학적(학교/전공/졸업여부), 학점(+학점 스케일 선택), 자격증(복수), 경력(회사/기간/직무/인턴여부, 복수), **기술스택(원티드 태그 API 기반 드롭다운/자동완성)**

### 주요 흐름
1. 이메일/비밀번호로 Supabase Auth 가입
2. 온보딩 폼으로 학적·학점·자격증·경력·스킬 입력 ("건너뛰기" 허용, 단 충족률 기능은 미완성 시 제한됨을 안내)
3. 완료 시 마이페이지로 이동

### 데이터 검증 규칙
- 학점 스케일(4.3/4.5 등) 필드 별도 필요
- 자격증/경력/스킬은 배열 형태의 동적 폼 필드

## 4. 로그인/회원가입 페이지 (기업용)

### 목적
가입 폼에서 "구직자/기업"을 선택하게 하고, 기업 선택 시 인재상 정보를 수집한다.

### 입력 항목
아이디, 비밀번호, 회사명, 원하는 인재상:
- 학점 기준 (`preferred_gpa_min`)
- 기술스택 (`preferred_skills[]`, **원티드 태그 API 기반 선택**)
- 유사 경험 유형 (`preferred_experience_type[]`)
- 인턴십 경험 요구 여부 (`internship_required`)

### MVP 범위의 명시적 생략 사항
기업 실체 인증(사업자등록증 검증 등)은 생략. 화면에 "MVP 단계로 별도 인증 없이 자체 신고 기반으로 운영됩니다" 문구 명시.

### 데이터 모델
```
company_profile: user_id, company_name, preferred_gpa_min,
  preferred_skills[], preferred_experience_type[], internship_required(boolean)
```

## 5. 마이페이지 (구직자용)

### 목적
경력·자격증 정보를 확인/수정하고, 지원현황·찜 목록을 모아본다.

### 주요 구성 요소
| 섹션 | 내용 | 데이터 출처 |
|---|---|---|
| 프로필 요약 | 학적, 학점, 자격증, 경력, 스킬 | Supabase (섹션 3 입력값) |
| 정보 수정 | 회원가입 폼과 동일 필드 재사용 | Supabase update |
| 나의 지원현황 | 지원한 공고 리스트 (자진신고 방식) | Supabase `applications` + 원티드 포지션 API 재조회 |
| 찜 목록 | 찜한 공고 리스트 | Supabase `bookmarks` + 포지션 API 재조회 |
| **전체 충족률 요약** | 지금까지 확인한 공고 중 충족률 상위/하위 스킬 통계 | 자체 로직 (1-1 계산 결과 누적) |

### 지원현황 기록 방식
섹션 7(기업상세 페이지)에서 "지원하기" 클릭 → 외부 이동 → 복귀 시 확인 모달 → Yes 선택 시 `applications`에 기록.

## 6. 마이페이지 (기업용)

### 목적
자사가 입력한 인재상 정보를 확인/수정하고, 지금까지 확인한 지원자 이력을 모아본다.

### 주요 구성 요소
| 섹션 | 내용 |
|---|---|
| 인재상 프로필 | 회원가입 시 입력한 기준 (수정 가능) |
| 확인한 지원자 로그 | 섹션 2에서 열람한 지원자 목록 히스토리 (충족률 포함) |

## 7. 기업별 상세정보 페이지

### 목적
구직자가 특정 공고를 자세히 보고, 실제 지원 여부를 결정하는 화면.

### 주요 구성 요소
| 요소 | 설명 | 데이터 출처 |
|---|---|---|
| 회사 소개 | 로고, 소개 이미지, 설명 | 원티드 회사(Company) API |
| 사내 복지/문화 태그 | company_tags 기반 | 원티드 회사 API |
| 모집기간 | 마감일 | 원티드 포지션 API (`due_time`) |
| 회사 평균연봉 (선택 표시) | 사업자번호 확보 시 | 원티드 회사/통계 API (제공 범위 내) |
| **요구사항 충족률 상세** | 어떤 항목을 충족/미충족했는지 세부 리스트 | 자체 로직 (1-1) |
| **지원하기 버튼** | 클릭 시 원티드 실제 지원 페이지로 새 탭 이동 | `job.url` |
| 지원자 통계 | ❌ 미제공 (통계 API는 실제 클라이언트 소유 공고에만 적용) |

### 지원 완료 판정 플로우
1. "원티드에서 지원하기 ↗" 클릭 → `window.open(job.url)`
2. 탭 복귀 시(`visibilitychange`) → "지원을 완료하셨나요?" 모달
3. Yes → `applications` 기록 / No·무응답 → 기록하지 않음

### UX 주의사항
버튼 라벨과 이동 시점에 "외부 사이트로 이동함"을 명확히 표시.

## 8. 지원서 작성 후 피드백 페이지

### 목적
구직자가 자기소개서를 입력하면, 목표 공고 대비 강약점과 경쟁자 비교 피드백을 제공한다. 이 데이터는 기업의 인재상 매칭(섹션 2)에도 함께 활용된다.

### 입력
- 자기소개서 텍스트
- 목표 공고 선택 (원티드 포지션 API 목록 중 하나)

### 출력 (LLM 기반 분석 — 원티드 API 범위 밖 영역)
| 항목 | 산출 방식 |
|---|---|
| 피드백 | 자소서 + 공고 요구사항 텍스트를 LLM에 함께 넣어 "부합하는 점 / 보완할 점" 생성 |
| 나의 장단점 | 이력서(경력·자격증) + 자소서를 LLM으로 구조화 분석 |
| **요구사항 충족률과의 연계** | 1-1에서 계산된 미충족 항목을 자소서 피드백에서 우선적으로 짚어주도록 프롬프트에 포함 (예: "인턴 경험 미충족 → 관련 경험을 자소서에서 어떻게 보완할지 제안") |
| 경쟁자 비교분석 | 같은 공고에 자소서를 제출한 다른 구직자들과 비교. 최소 표본(예: 5명) 미만이면 "비교 데이터 부족"으로 대체 |

### 데이터 흐름과 기업 측 연결
사용자가 제출 시 공유 동의하면, 해당 자소서 요약이 섹션 2(기업용 랜딩페이지)의 매칭 대상에 포함된다. **동의 체크박스 필수.**

### 데이터 모델
```
self_intros: user_id, job_id, content, submitted_at, shared_with_company(boolean)
feedback_results: self_intro_id, strengths[], improvements[], generated_at
```

```
  - user_id, company_name, preferred_gpa_min, preferred_skills[],

  - user_id, job_id, applied_at, status ('self_reported')

  - user_id, job_id, created_at

  - user_id, job_id, content, submitted_at, shared_with_company

feedback_results
  - self_intro_id, strengths[], improvements[], generated_at
```
